* Flow of program till now (for user registration and auth) - When a user first visits your single-page application, the root App.jsx component mounts and immediately triggers getCurrentUser(). Because the user is new and lacks an authentication cookie, the backend's isAuth middleware intercepts the request, finds no token, and short-circuits the operation by returning a 401 Unauthorized response, prompting your frontend to keep or redirect the user on the /auth screen. Once the user clicks "Continue with Google" and finishes the Firebase popup authentication, the frontend sends their profile data to your /api/auth/google backend endpoint. The server detects that the user does not yet exist in MongoDB, creates a brand-new user document—which automatically generates a unique _id—encodes that fresh ID into a secure JWT token, and drops it right into the user's browser cookies. From this point forward, whenever the application mounts or calls getCurrentUser(), the browser automatically bundles this cookie with the network request, allowing the backend middleware to decode it, extract the userId, and grant seamless access to the controller.
